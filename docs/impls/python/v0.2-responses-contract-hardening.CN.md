# Python v0.2 Responses Contract Hardening 设计方案

状态：实现设计  
日期：2026-05-06  
最近更新：2026-05-06

## 背景

Python v0.1 alpha 已完成基础 core 模型、OpenAI Responses API generation/streaming、function tool 映射、text embedding、usage 与 capability 基础表达。v0.2 的目标不是扩展新的 API 家族，而是在进入 v0.3 core API family expansion 之前，先偿还 OpenAI Responses API 合约债务。

本方案细化 [impls/python/evolution-plan.CN.md](impls/python/evolution-plan.CN.md) 中的 v0.2 阶段，约束后续实现应如何修改 core 与 OpenAI adapter。相关背景设计见 [design/provider-capability-integration.CN.md](design/provider-capability-integration.CN.md) 与 [impls/python/openai-adapter.CN.md](impls/python/openai-adapter.CN.md)。

本地开发环境当前验证到的 OpenAI Python SDK 版本为 `2.34.0`。v0.2 应以该版本的 Responses API surface 作为最低合约基线，再通过 fixture tests 固化映射行为。

## 设计目标

### 稳定第一阶段 public API

v0.2 必须保持现有 `OpenAIClient.generate()`、`stream_generate()`、`embed()` 及其 async 变体的基本调用方式不变。允许对 core dataclass 增加向后兼容字段，但不应重命名现有入口、不改变 provider client 初始化语义、不引入自动 routing 或自动工具执行。

### 让 Responses API 参数映射可验证

OpenAI adapter 必须明确区分三类参数：

- 已归一化的通用 generation 参数，例如 `temperature`、`top_p`、`max_output_tokens`、`reasoning`、`tool_choice`。
- OpenAI Responses API 当前明确支持的参数，例如 `text.format`、`parallel_tool_calls`、`stream`。
- 暂未归一化或 provider-specific 的参数，通过 `provider_options` 透传，例如 `store`、`previous_response_id`、`prompt_cache_key`、`include`、`service_tier`。

mapper 不应生成当前 SDK surface 中明显不匹配的参数。特别是 v0.1 中 `StreamOptions.include_usage` 被映射为 `stream_options.include_usage`，而 OpenAI Python SDK `2.34.0` 的 Responses `stream_options` 只暴露 `include_obfuscation`。v0.2 应修正这个映射。

### 让 streaming 可重放、可累积、可诊断

OpenAI Responses streaming 不是只有 text delta。v0.2 应覆盖 response lifecycle、output item lifecycle、content part lifecycle、text delta/done、function call arguments delta/done、reasoning summary/text 事件、failed/incomplete/error，以及未知事件的 raw passthrough。

normalized stream event 必须保留足够 metadata，让用户能够：

- 按 provider 原始顺序处理事件。
- 从事件流重建最终 `GenerationResponse`。
- 诊断 provider 新增事件或无法归一化的事件。

### 明确 structured output 的通用模型

Responses API 使用 `text.format` 表达 JSON Schema structured output。`vatbrain` 不兼容 JSON mode / `json_object` 调用方式；`ResponseFormat` 只表达 JSON Schema structured output。

### 改善 provider error mapping

v0.2 至少应稳定以下错误层次：

- request error：provider SDK/API 调用失败。
- response mapping error：provider response 或 stream event 无法映射为 vatbrain 模型。
- unsupported capability：用户请求了 adapter 已知不能支持的能力。

错误对象应尽量保留 provider、status code、request id、error type、param/code、raw/cause 等诊断信息，同时保持现有异常类型可捕获。

## 非目标

- 不新增 Volcengine adapter。
- 不新增 resource/file API。
- 不新增 media generation API。
- 不扩展 multimodal/sparse embedding。
- 不完整建模 provider-hosted tools。
- 不引入 Chat Completions API fallback。
- 不改变 Full-context First 编程模型。
- 不自动执行 function tools。

## 模块职责

### core.generation

`core.generation` 负责表达 provider-neutral generation request、response 与 normalized stream event。v0.2 可在此处做有限扩展，并收敛已不再推荐的旧 structured output 入口：

- 收敛 `ResponseFormat`，明确 JSON Schema structured output 的 `name`、`description`、`schema`、`strict` 语义。
- 扩展 `StreamEventType`，补充 content part、text done、reasoning、incomplete、error 等事件类型。
- 扩展 `GenerationStreamEvent.metadata` 的约定字段，例如 `provider_event_type`、`output_index`、`content_index`、`sequence_number`、`status`。
- 增加 `GenerationStreamAccumulator`，从 normalized events 重建 `GenerationResponse`。

v0.2 不应在 `core.generation` 中加入 `RemoteContextHint`。`previous_response_id`、`store`、`prompt_cache_key` 等仍通过 `provider_options` 显式透传，v0.3 再统一建模。

### core.errors

`core.errors` 负责提供稳定的异常层次。v0.2 应保持现有异常类名，同时增强 `ProviderRequestError` 与 `ProviderResponseMappingError` 的诊断字段。

建议字段：

```text
provider
operation
status_code
request_id
error_type
error_code
error_param
raw
cause
```

`UnsupportedCapabilityError` 应用于 adapter 知道无法支持的请求，例如 OpenAI v0.2 不支持通过通用 `StreamOptions.include_usage` 生成 `stream_options.include_usage`。

### providers.openai.mapper

`providers.openai.mapper` 负责 request/response 的非流式映射。v0.2 的重点是 Responses create params、structured output、usage 与 output item 的合约加固。

该模块不得引入 Chat Completions API 映射路径。

### providers.openai.stream

`providers.openai.stream` 负责 provider stream event 到 `GenerationStreamEvent` 的映射。它必须保留 raw event，并尽可能填充 normalized 字段与 metadata。

不应在 stream mapper 中执行工具、不应隐式发起下一轮 generation、不应把 provider-hosted tool 的生命周期强行伪装成 user function tool。

### providers.openai.client

`OpenAIClient` 负责 provider SDK 调用、错误包装和 sync/async 入口。v0.2 应保持入口签名基本稳定，并确保 streaming request 使用 Responses API `stream=True`。

### pyproject

`python/pyproject.toml` 应把 `openai` 依赖下限调整为实际验证过的 Responses API SDK 范围。v0.2 建议使用：

```text
openai>=2.34.0,<3
```

如果实现阶段验证了更低版本同样满足 Responses API 合约，可以下调下限，但必须有测试或明确记录支撑。

## 参数映射设计

### Generation create params

`to_openai_generation_params(request, stream=False)` 的输出应满足以下规则：

```text
GenerationRequest.model -> model
GenerationRequest.items -> input
GenerationRequest.tools -> tools
GenerationConfig.temperature -> temperature
GenerationConfig.top_p -> top_p
GenerationConfig.max_output_tokens -> max_output_tokens
GenerationConfig.stop -> stop
ReasoningConfig -> reasoning
ToolCallConfig.parallel_tool_calls -> parallel_tool_calls
ToolCallConfig.tool_choice -> tool_choice
ResponseFormat -> text.format
stream=True -> stream=True
provider_options -> top-level OpenAI params, with explicit override semantics
```

`provider_options` 保持最后合并，因此用户可以在必要时覆盖 adapter 默认映射。测试应覆盖这个 override 行为。

### StreamOptions

`StreamOptions.include_usage` 是 vatbrain 的通用 streaming option，但 OpenAI Responses API 在当前 SDK baseline 下不使用 `stream_options.include_usage`。

v0.2 的 OpenAI adapter 行为：

- `StreamOptions(include_usage=None)`：不生成 `stream_options`。
- `StreamOptions(include_usage=True)`：不生成 OpenAI `stream_options.include_usage`；usage 通过 `response.completed`、`response.incomplete` 或 provider 原始 usage 事件映射。
- `StreamOptions(include_usage=False)`：不尝试压制 provider final response usage。
- OpenAI provider-specific stream options 通过 `provider_options={"stream_options": {"include_obfuscation": ...}}` 透传。

如果后续 OpenAI SDK/API 新增可验证的 usage stream option，再通过 fixture test 明确启用。

### Structured Output

v0.2 应将 `ResponseFormat` 的 JSON schema 结构明确为：

```text
ResponseFormat
- json_schema: dict
- json_schema_name: str | None
- json_schema_description: str | None
- json_schema_strict: bool | None
```

映射策略：

- `ResponseFormat(json_schema={...})` 默认把 `json_schema` 视为实际 schema，并使用默认 name `response`，除非用户显式设置 `json_schema_name`。
- 不提供 `type`/mode 选择器，不接受 `json_object` 或 provider wrapper 形态。
- `strict` 只有在用户显式设置时才输出；不由 adapter 隐式改成 `True`。

目标 OpenAI 参数形态：

```text
text:
  format:
    type: json_schema
    name: ...
    description: ...
    schema: ...
    strict: ...
```

对于模型是否支持 structured output，v0.2 不维护内部权威模型表。provider 返回不支持时由 `ProviderRequestError` 包装。

## Response 映射设计

### Output item

v0.2 继续只稳定映射当前 core 能表达的输出：

- OpenAI `message` -> `MessageItem(role=assistant, parts=[TextPart, ...])`
- OpenAI `function_call` -> `FunctionCallItem`

遇到 core 尚不能表达的 output item，例如 reasoning item、provider-hosted tool item、audio item，应采用以下策略：

- 非流式 response：先保留 raw，并在 `GenerationResponse.metadata["unsupported_output_items"]` 中记录摘要；如果该 item 是唯一输出且无法给出有意义 normalized output，可抛出 `ProviderResponseMappingError`。
- 流式事件：不抛出，映射为 `StreamEventType.UNKNOWN` 或 reasoning/raw passthrough 事件，并保留 `raw_event`。

这样可以避免 provider 新增输出类型直接破坏 streaming 消费者，同时让非流式响应保持可诊断。

### Usage

`usage_from_openai()` 应继续支持 Responses API 与兼容 OpenAI-like usage 对象：

```text
input_tokens / prompt_tokens
output_tokens / completion_tokens
total_tokens
input_tokens_details.cached_tokens
output_tokens_details.reasoning_tokens
```

无法归一化的 provider usage 信息保留到 `Usage.raw`。

## Streaming 映射设计

### Event metadata 约定

所有 OpenAI stream events 都应写入：

```text
metadata["provider_event_type"]
metadata["sequence_number"]      # provider event 自带时写入
metadata["output_index"]         # provider event 自带时写入
metadata["content_index"]        # provider event 自带时写入
metadata["status"]               # lifecycle event 自带时写入
```

`GenerationStreamEvent.sequence` 是 vatbrain adapter 本地枚举顺序。若 provider event 自带 `sequence_number`，二者都保留，避免混淆。

### Lifecycle events

建议 normalized mapping：

```text
response.created -> RESPONSE_CREATED
response.in_progress -> RESPONSE_STARTED
response.queued -> RESPONSE_STARTED, metadata.status="queued"
response.completed -> RESPONSE_COMPLETED
response.incomplete -> RESPONSE_INCOMPLETE
response.failed -> RESPONSE_FAILED
response.error / error -> RESPONSE_ERROR
```

`response.completed` 和 `response.incomplete` 若携带完整 response，应调用 `from_openai_generation_response()`；若 mapping 失败，返回带 raw response 的降级 `GenerationResponse`，并在 metadata 中记录 mapping error。

### Output item lifecycle

```text
response.output_item.added -> ITEM_CREATED
response.output_item.done -> ITEM_COMPLETED
```

当 output item 为 `function_call` 时，应尽早填充 `FunctionCallItem`，即使 arguments 仍为空或未完成。

### Content part lifecycle

```text
response.content_part.added -> CONTENT_PART_CREATED
response.content_part.done -> CONTENT_PART_COMPLETED
```

content part 的 `output_index`、`content_index` 必须进入 metadata。若 content part 当前能表达为 `TextPart`，可放入 `delta` 或 `item` 的 metadata；无法表达则 raw passthrough。

### Text delta/done

```text
response.output_text.delta -> TEXT_DELTA
response.output_text.done -> TEXT_COMPLETED
```

`delta` 字段放增量文本。`done` 事件的完整文本放入 `delta` 或 metadata 中的 `text`，实现时需选择一种固定约定并测试。

历史兼容：如果现有用户依赖 `ITEM_DELTA`，可在 v0.2 保留 `ITEM_DELTA` 作为 alias 或通过 `metadata["semantic_type"]="text_delta"` 过渡。最终推荐使用更明确的 `TEXT_DELTA`。

### Function call arguments

```text
response.function_call_arguments.delta -> TOOL_CALL_DELTA
response.function_call_arguments.done -> TOOL_CALL_COMPLETED
```

`delta` 保存 arguments 增量或最终 arguments 字符串。`name`、`call_id`、`output_index`、`item_id` 应尽量保留到 metadata 或 `FunctionCallItem`。

OpenAI 官方 function calling streaming 指南说明，function call arguments 需要按 `output_index` 聚合 delta 后得到最终 JSON arguments，因此 accumulator 必须以 `output_index` 优先定位工具调用。

### Reasoning events

v0.2 不新增 `ReasoningItem`。OpenAI reasoning 相关事件采用 raw passthrough 加初步 normalized event：

```text
response.reasoning_summary_part.added -> REASONING_CREATED
response.reasoning_summary_text.delta -> REASONING_DELTA
response.reasoning_summary_text.done -> REASONING_COMPLETED
response.reasoning_summary_part.done -> REASONING_COMPLETED
response.reasoning_text.delta -> REASONING_DELTA
response.reasoning_text.done -> REASONING_COMPLETED
```

reasoning 文本或 summary 文本放入 `delta`。metadata 应标明 `reasoning_kind`，例如 `summary` 或 `text`。v0.3 再把 reasoning 输出提升为 `ReasoningItem`。

### Unknown/provider-hosted tool events

OpenAI SDK 可能返回 file search、web search、image generation、MCP、code interpreter、audio/refusal 等事件。v0.2 对这些事件不做完整 core 建模，但必须：

- 返回 `StreamEventType.UNKNOWN` 或最接近的 lifecycle event。
- 设置 `metadata["provider_event_type"]`。
- 保留 `raw_event`。
- 不因未知事件中断 stream iteration。

## Stream Accumulator 设计

新增 `GenerationStreamAccumulator`，推荐放在 `core.generation`。它只消费 normalized `GenerationStreamEvent`，不依赖 OpenAI SDK 类型。

概念 API：

```python
accumulator = GenerationStreamAccumulator(provider="openai")

for event in client.stream_generate(...):
    accumulator.add(event)

response = accumulator.to_response()
```

行为规则：

- 若收到 `event.response` 且事件为 completed/incomplete，优先使用该 response 作为最终 response。
- 若没有完整 response，则根据 item lifecycle 与 delta 重建最小 `GenerationResponse`。
- 文本输出按 `output_index` 和 `content_index` 聚合。
- function call arguments 按 `output_index` 优先聚合，缺失时回退到 `item_id`。
- usage 使用最近一次非空 `event.usage`。
- response id/model/status 从 lifecycle event 或 final response 中提取。
- failed/error 事件不自动抛异常，但 `to_response()` 应在 metadata 中记录 terminal error。

限制：

- v0.2 accumulator 不重建 reasoning item。
- v0.2 accumulator 不重建 provider-hosted tool item。
- v0.2 accumulator 不执行 function tool。

## 错误映射设计

### Request error

`OpenAIClient` 捕获 provider SDK 抛出的异常后，应包装为 `ProviderRequestError`。包装时通过宽松 introspection 提取：

```text
status_code
request_id
type
code
param
message
response/body
```

不得强依赖某个 OpenAI SDK 私有异常类，以降低 SDK patch version 变化的影响。

### Response mapping error

`from_openai_generation_response()` 处理未知 output item 时，应提供带 provider item type 的错误信息。streaming mapper 对未知 event 不抛 mapping error，但应在 metadata 中记录。

### Unsupported capability

当 mapper 在本地即可确定请求不支持时，抛出 `UnsupportedCapabilityError`，例如：

- OpenAI text embedding mapper 收到 image/audio/video embedding input。
- 通用 stream option 无法映射且语义不能安全忽略。

对于 provider/model 支持情况不确定的字段，不在本地猜测阻断，而让 provider 返回错误，再包装为 `ProviderRequestError`。

## 实现步骤

### Step 1: SDK contract audit

- 调整 `pyproject.toml` 的 `openai` 依赖范围。
- 修正 `stream_options.include_usage` 映射。
- 增加 Responses create params fixture tests。
- 固定 `provider_options` override 规则。

### Step 2: Structured output hardening

- 将 `ResponseFormat` 收敛为 JSON Schema structured output 专用模型。
- 重写 `_response_format_to_openai_text()`。
- 增加 JSON Schema canonical mapping、JSON mode rejection 和 provider wrapper rejection tests。

### Step 3: Streaming event hardening

- 扩展 `StreamEventType`。
- 系统补齐 OpenAI stream event mapper。
- 所有 event 保留 `provider_event_type` 与 raw event。
- 增加 stream event fixture tests。

### Step 4: Accumulator

- 新增 `GenerationStreamAccumulator`。
- 实现 text/function call 的最小重建。
- 增加 accumulator tests。

### Step 5: Error mapping

- 增强 error dataclass-like fields。
- 在 client 中统一包装 OpenAI SDK exceptions。
- 增加 request/mapping/unsupported capability tests。

### Step 6: Documentation and status

- 更新 [impls/python/STATUS.md](impls/python/STATUS.md)。
- 如 public API 新增字段影响用户用法，更新 [user/python/quickstart.CN.md](user/python/quickstart.CN.md) 与 [user/python/STATUS.md](user/python/STATUS.md)。

## 验收标准

- `python/.venv/bin/python -m pytest` 通过。
- `to_openai_generation_params()` 不再为 OpenAI Responses API 生成未验证的 `stream_options.include_usage`。
- structured output 映射明确输出 `text.format`。
- OpenAI streaming mapper 覆盖 v0.2 指定事件，并对未知事件 raw passthrough。
- `GenerationStreamAccumulator` 能从 text 与 function call stream fixtures 重建 `GenerationResponse`。
- OpenAI SDK request errors 被包装为可诊断的 `ProviderRequestError`。
- v0.1 的基础调用示例继续可用。

## 风险与决策点

### 是否在 v0.2 引入 ResponseFormat 新字段

为正确表达 JSON Schema structured output，v0.2 引入 `json_schema_name`、`json_schema_description`、`json_schema_strict`，并移除 JSON mode / `json_object` 兼容入口。`ResponseFormat` 不再承担多模式选择职责。

### Unknown output item 的非流式策略

如果 provider 返回 reasoning 或 hosted tool item，v0.2 core 尚不能完整表达。实现时需要在“尽量返回可用 response”和“避免隐藏语义损失”之间取舍。本方案建议：有可用 message/function_call 时返回并记录 metadata；完全无法表达时抛 mapping error。

### OpenAI SDK 下限

当前本地验证版本是 `2.34.0`。若实现阶段希望支持更低版本，需要用同一组 mapper/stream fixture tests 验证 Responses API 参数和事件类型，不应只根据宽泛版本号假设兼容。
