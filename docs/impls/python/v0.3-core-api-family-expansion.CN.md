# Python v0.3 Core API Family Expansion 实现基线

状态：已实现
日期：2026-05-06
最近更新：2026-05-13

## 定位

v0.3 是 Python 参考实现的核心模型稳定节点。它将 `vatbrain` 从“OpenAI Responses API adapter 的第一阶段封装”推进为“可承载多 provider、多 API family 的稳定 core 抽象”。后续 v0.4 及以后主要应围绕新增 provider adapter 展开，而不是继续在 OpenAI adapter 内堆积临时结构。

本文件记录 v0.3 已落地的 implementation baseline，供后续开发者实现 Volcengine、Anthropic、Google 等 provider 时对照。高层语义见 [design/high-level-design.CN.md](design/high-level-design.CN.md)、[design/provider-capability-integration.CN.md](design/provider-capability-integration.CN.md) 和 [design/provider-native-replay.CN.md](design/provider-native-replay.CN.md)。用户侧用法见 [user/python/quickstart.CN.md](user/python/quickstart.CN.md) 与 [user/python/api-reference.CN.md](user/python/api-reference.CN.md)。

## 设计原则

### Full-context First

`GenerationRequest.items` 始终是一次 generation 调用的完整语义上下文。Provider 的 `previous_response_id`、store、cache 或 future conversation/context 能力只能作为优化 hint，不是 `vatbrain` 的核心状态。

Full-context First 不要求 provider 请求层每次都全量传输。当用户通过 `RemoteContextHint.covered_item_count` 明确说明 `previous_response_id` 已覆盖完整 `items` 的历史前缀时，provider adapter 可以只发送未覆盖 suffix。若 remote context 失效并触发显式 fallback，client 必须重新基于完整 `items` 构造请求。

### API Family Separation

v0.3 将 core 拆分为若干 API family：

- `generation`：多轮推理、结构化输出、工具调用、reasoning、流式事件。
- `embeddings`：样本级向量化、多模态输入、dense/sparse vector。
- `resources`：provider 文件资源生命周期。
- `media`：图片/视频等媒体生成请求、产物、任务、流式事件。
- `tools`：用户代码执行的 function/custom tool 协议。
- `capabilities`：adapter/model 能力描述。

这些 API family 共享 `Item`、`Usage`、error、provider client，但不能被合并为一个万能 request。

### Provider-native Replay

Normalized `Item` 负责跨 provider 编程模型；`ProviderItemSnapshot` 负责同 provider/API family 高保真重放。后续 provider adapter 从 provider response 映射 output item 时，应尽量保存可重放原始 payload。跨 provider replay 暂不支持。

### Explicit Tool Ownership

v0.3 只抽象用户代码执行的 function/custom tool。Provider-hosted tool、remote tool、MCP tool 暂不进入通用 core 抽象，也不在用户文档中暴露为通用能力。

## 模块基线

### core.items

文件：`python/whero/vatbrain/core/items.py`

已实现：

- `Role(system | developer | user | assistant | tool)`
- `ItemKind(message | function_call | function_result | reasoning)`
- `ItemPurpose(instruction | query | context | answer | tool_io | artifact)`
- `PartKind(text | image | audio | video | file)`
- `AssistantMessagePhase(commentary | final_answer)`
- `ProviderItemSnapshot`
- `TextPart`
- `ImagePart`
- `AudioPart`
- `VideoPart`
- `FilePart`
- `MessageItem`
- `FunctionCallItem`
- `FunctionResultItem`
- `ReasoningItem`
- `provider_snapshot_key()`
- `provider_snapshot_for()`

实现语义：

- `MessageItem` 是 message-like item，可包含多个 content part。
- `MessageItem.system/developer/user/assistant()` 是用户侧便捷构造。
- `assistant_phase` 只允许出现在 assistant message 上。OpenAI adapter 将其映射为 Responses API `phase`。
- `ImagePart` 需要且只能提供 `url` 或 `data`。
- `AudioPart`、`VideoPart`、`FilePart` 的 `file_id`、`url`、`data`、`local_path` 互斥；`local_path` 只是 metadata，不触发本地 I/O 或上传。
- `FunctionCallItem.type` 支持 `function` 与 `custom`。Custom tool 的 raw input 保存于 `input`；为兼容旧路径，也会同步保存在 `arguments`。
- `FunctionResultItem.tool_type` 用于回填 custom tool 结果，使 OpenAI adapter 能映射为 `custom_tool_call_output`。
- `ProviderItemSnapshot` 挂载于 `Item.provider_snapshots`，不放入 metadata。Snapshot 只用于同 provider/API family replay。

### core.generation

文件：`python/whero/vatbrain/core/generation.py`

已实现：

- `GenerationConfig`
- `ResponseFormat`
- `ReasoningConfig`
- `RemoteContextHint`
- `ReplayMode`
- `RemoteContextInvalidBehavior`
- `ReplayPolicy`
- `ToolCallConfig`
- `StreamOptions`
- `GenerationRequest`
- `GenerationResponse`
- `StreamEventType`
- `GenerationStreamEvent`
- `GenerationStreamAccumulator`

实现语义：

- `GenerationRequest.items` 不可为空，并表示完整上下文。
- `ResponseFormat` 只接受 JSON Schema body；如果传入 provider wrapper 会抛错。v0.3 不兼容 JSON mode / `json_object`。
- `ReasoningConfig.effort` 保持字符串，具体 provider 支持的 effort 由 capability 声明，不写死为统一枚举。
- `RemoteContextHint.previous_response_id` 表达 provider previous response；`covered_item_count` 表达其覆盖完整 `items` 的前缀长度；`store` 表达本轮 response 是否请求 provider 存储。
- `ReplayPolicy.mode` 支持 `normalized_only`、`prefer_provider_native`、`require_provider_native`。
- `ReplayPolicy.on_remote_context_invalid` 默认 `raise`；仅当用户显式设置 `replay_without_remote_context` 时，client 才能在 previous response 失效后移除 remote context 并用完整 `items` 重试一次。
- `ReplayPolicy.cross_provider` 当前只允许 `unsupported`。
- `GenerationStreamAccumulator` 可从标准化 stream event 重建文本输出与 function/custom tool call 输出。

OpenAI 差分传输规则已落地：

- 存在 `previous_response_id` 时必须提供 `covered_item_count`，否则 OpenAI mapper 抛出清晰错误。
- `covered_item_count > len(items)` 非法。
- suffix 为空时 OpenAI mapper 抛错，因为 Responses API 本轮 input 需要至少一个新增 item。
- optimized attempt 发送 suffix。
- fallback attempt 重新构造完整 input，并移除 `previous_response_id`。

### core.tools

文件：`python/whero/vatbrain/core/tools.py`

已实现：

- `ToolChoice(auto | none | required)`
- `ToolExecutionOwner(user)`
- `FunctionToolType(function | custom)`
- `FunctionToolSpec`
- `ToolSpec = FunctionToolSpec`

实现语义：

- `ToolSpec` 是用户侧主入口，保留第一阶段兼容。
- `ToolSpec(type="function")` 是默认 function tool，`parameters_schema` 映射为 JSON Schema object。空 `parameters_schema` 表示普通 function tool 的空 object 参数。
- `ToolSpec(type="custom")` 表达 raw string input 工具。OpenAI adapter 映射为 custom tool，不映射 `parameters_schema` / `strict`。
- `execution_owner` 当前只能是 `user`。Provider-hosted/remote/MCP tool 后续另行设计。

### core.embeddings

文件：`python/whero/vatbrain/core/embeddings.py`

已实现：

- `EmbeddingInput`
- `SparseEmbedding`
- `EmbeddingRequest`
- `EmbeddingVector`
- `EmbeddingResponse`

实现语义：

- `EmbeddingInput` 表达一个待向量化样本，可由字符串、content parts 或 message parts 构造。
- `EmbeddingRequest.instructions` 是 embedding 通用语义，用于 query/corpus/multimodal retrieval 等任务提示。
- `EmbeddingRequest.sparse_embedding` 请求 sparse vector。
- `EmbeddingVector.dense` 与 `EmbeddingVector.sparse` 分别表达 dense/sparse 结果；`embedding` 保留旧用法兼容。
- OpenAI adapter 当前仅支持 text embedding；多模态/sparse 表达主要用于后续 provider。

### core.resources

文件：`python/whero/vatbrain/core/resources.py`

已实现：

- `FileStatus(uploaded | processing | ready | failed | deleted | expired | unknown)`
- `FilePurpose(assistants | batch | fine_tune | vision | retrieval | media | other)`
- `FilePreprocessConfig`
- `FileUploadRequest`
- `FileResource`

实现语义：

- Core 只定义文件资源模型，不执行本地文件 I/O。
- `FileUploadRequest.file` 可是 bytes、路径字符串、`PathLike` 或 provider adapter 支持的文件对象。
- `FileResource` 表达 provider 管理的文件资源，可被 `FilePart(file_id=...)` 引用。
- 当前 OpenAI adapter 尚未暴露 file/resource 方法。

### core.media

文件：`python/whero/vatbrain/core/media.py`

已实现：

- `MediaKind(image | video | audio)`
- `TaskStatus(queued | running | completed | failed | canceled | expired | unknown)`
- `MediaArtifact`
- `ImageGenerationRequest`
- `ImageGenerationResponse`
- `ImageGenerationStreamEvent`
- `MediaGenerationTask`

实现语义：

- Media generation 是独立 API family，不复用 `GenerationRequest`。
- `MediaArtifact` 至少需要 `url`、`data`、`file_id` 或 `raw` 之一。
- `ImageGenerationRequest.input_items` 可承载参考图片、文件等上下文。
- `MediaGenerationTask` 表达视频等异步生成任务。
- 当前 OpenAI adapter 尚未暴露 image/video generation 方法。

### core.errors / usage / client

已实现：

- `ClientConfig`

- `VatbrainError`
- `InvalidItemError`
- `UnsupportedCapabilityError`
- `ProviderErrorDetails`

实现语义：

## OpenAI Adapter 基线

主要文件：

- `python/whero/vatbrain/providers/openai/client.py`
- `python/whero/vatbrain/providers/openai/mapper.py`
- `python/whero/vatbrain/providers/openai/stream.py`
- `python/whero/vatbrain/providers/openai/capabilities.py`

已支持：

- Responses API generation。
- Responses API streaming。
- Async generation / streaming。
- JSON Schema structured output via `text.format`。
- Pydantic helper convenience：`generate_parsed()` / `agenerate_parsed()`。
- Function tool 映射。
- Custom tool 映射。
- Tool call output 回填。
- Text/image message input 的基础映射。
- Reasoning config 参数透传映射。
- Remote context：`previous_response_id`、`store`。
- `covered_item_count` 驱动的 previous response 差分传输。
- previous response 失效时的显式 fallback replay。
- Provider-native item snapshot replay。
- OpenAI assistant message `phase` 与 `AssistantMessagePhase` 映射。
- Streaming lifecycle、content part、text、function/custom tool call、reasoning、usage、error、unknown passthrough。
- Text embedding。
- Adapter/model capability 查询与用户覆写。

暂未支持：

- Chat Completions fallback。
- 自动工具执行或 agent loop。
- Provider-hosted/remote/MCP tool 的通用抽象。
- Provider conversation 持久化上下文抽象。
- File/resource API 方法。
- 多模态 embedding。
- Image/video generation 方法。
- 跨 provider replay。

## Pydantic Structured Output 基线

主要文件：

- `python/whero/vatbrain/structured.py`
- `python/whero/vatbrain/providers/openai/client.py`

已实现：

- `pydantic_output(output_type, name=None, description=None, strict=True)`
- `PydanticOutputSpec`
- `ParsedGenerationResponse`
- `StructuredOutputParseError`
- OpenAI client `generate_parsed()` / `agenerate_parsed()`

实现语义：

- Helper 依赖 Pydantic v2，可通过 optional extra `whero-vatbrain[pydantic]` 安装。
- Helper 生成普通 `ResponseFormat`，不绕过 provider adapter。
- 默认 schema name 来自 type `__name__`，并清理为 provider-safe 名称。
- 默认 schema description 来自 type docstring。
- 默认 `strict=True`，会将 Pydantic schema 调整为更适合 structured output 的 strict schema。
- `parse_response()` 从 assistant text 中提取 JSON 并用 Pydantic `TypeAdapter.validate_json()` 解析。
- 当前不做 partial JSON 流式解析。

## 后续 Provider Adapter 指南

实现新 provider 时应遵守：

- Provider client 以 provider 为初始化粒度，每次调用显式传入 model。
- 对同时支持 Responses API 与 Chat Completions API 的 provider，generation adapter 应优先且仅使用 Responses 风格 API。Chat API 只作为语义对照。
- 先声明 adapter capability，再实现 mapper。
- 请求 mapper 不应猜测 history/append 边界；只有 `RemoteContextHint.covered_item_count` 能驱动差分传输。
- Response mapper 应保存可重放 provider snapshot。
- 若 provider 原生字段具备跨 provider 潜力，可评估提升为 core 抽象；否则保留在 snapshot/raw/provider_options。
- 不隐式读取或上传 `local_path`。
- 不自动执行工具，不自动发起 follow-up。
- Provider-specific 高级参数先放入 `provider_options`，只有具备跨 provider 语义后才进入 core。

## 与 v0.4 的关系

v0.4 的主要目标是新增 Volcengine adapter MVP，用第二个 provider 验证 v0.3 core 的表达能力。预期覆盖：

- Responses generation/streaming。
- Files API。
- Multimodal embedding。
- Reasoning config/output。
- Function tools。
- Remote context/store hint。
- Capability 声明。

Hosted tools、image/video generation、provider conversation 持久化上下文、跨 provider replay 仍不进入 v0.4 MVP，除非另行设计并更新知识库。
