# Python v0.3 Core API Family Expansion 实现方案

状态：实现设计  
日期：2026-05-06  
最近更新：2026-05-06

## 背景

Python v0.2 已完成 OpenAI Responses API 合约加固，稳定了 generation 参数映射、structured output、streaming event、stream accumulator 与 provider error diagnostics。v0.3 的目标是扩展 Python core 的领域模型，使其能表达 provider 能力整合设计中的核心语义，并为 v0.4 Volcengine adapter MVP 做准备。

本方案细化 [impls/python/evolution-plan.CN.md](impls/python/evolution-plan.CN.md) 中的 v0.3 阶段。设计依据包括 [design/high-level-design.CN.md](design/high-level-design.CN.md)、[design/provider-capability-integration.CN.md](design/provider-capability-integration.CN.md)、[impls/python/v0.2-responses-contract-hardening.CN.md](impls/python/v0.2-responses-contract-hardening.CN.md) 与 [3rds/volengine/INDEX.md](3rds/volengine/INDEX.md)。

## 设计目标

### Core 先行

v0.3 只扩展 provider-neutral core 模型，不实现新的 provider adapter。新增模型应让 generation、embedding、resource/file、media generation、tools、capability 的 API family 边界在代码中成型。

### 兼容 v0.2 用户代码

现有 OpenAI 用法必须继续可用：

```python
client.generate(model="...", items=[MessageItem.user("hello")])
client.stream_generate(model="...", items=[MessageItem.user("hello")])
client.embed(model="text-embedding-3-small", inputs=["hello"])
ToolSpec(name="lookup", parameters_schema={...})
```

允许新增 dataclass 字段、枚举值和别名，但不应删除或重命名 v0.2 public symbols。

### 表达而不执行

v0.3 的资源、媒体和 function tool 模型只表达请求、响应、状态与责任，不执行本地文件上传、不下载媒体、不轮询任务、不自动执行工具、不隐式发起下一轮 generation。

### 为 Volcengine 留出映射锚点

v0.3 应优先覆盖火山方舟 v0.4 所需的 core 语义：

- video/image/file part。
- provider file id 与 file resource lifecycle。
- multimodal/sparse embedding。
- reasoning config 与 reasoning output item。
- remote context/cache/store hint，其中 remote context 暂只覆盖 previous response，不表达 provider conversation 持久化。
- user-executed function tool 兼容模型。
- media artifact/task。
- API family capability。

## 非目标

- 不实现 Volcengine API 调用。
- 不要求 OpenAI adapter 立刻支持所有新增 `Part` 或 `Item`。
- 不实现 file upload/list/delete client 方法。
- 不实现 image/video generation client 方法。
- 不自动上传 `local_path`。
- 不自动执行 function tool。
- 暂不抽象 provider-hosted tool、remote tool 或 MCP tool；这些能力日后另行设计。
- 不引入 provider routing、fallback 或自动模型选择。
- 不维护内部权威模型能力表。

## 模块职责

### core.items

`core.items` 继续作为跨 API family 的内容元模型，负责表达 message content、tool protocol item 与 reasoning item。


```text

- url

- local_path
- filename
- mime_type
- media_type

```

来源字段约束：

- `AudioPart`、`VideoPart`、`FilePart` 应至少提供一个来源：`file_id`、`url`、`data` 或 `local_path`。
- 可以允许同时提供 `file_id` 与 metadata，但不应同时提供多个实际内容来源并让 adapter 猜测优先级。
- `local_path` 只是 metadata，不触发本地 I/O。

新增 item：

```text
ReasoningItem
- text
- summary
- provider
- visibility
- can_be_replayed
- id
- status
- metadata
- raw
```

`ReasoningItem` 表达 provider 返回的 reasoning summary/text 或原始 reasoning 内容。由于 provider 对 reasoning 可见性和回传策略差异很大，`visibility` 与 `can_be_replayed` 必须显式存在，默认应保守。

兼容策略：

- `ContentPart = TextPart | ImagePart | AudioPart | VideoPart | FilePart`。
- `Item = MessageItem | FunctionCallItem | FunctionResultItem | ReasoningItem`。
- `MessageItem.user("text")` 等快捷构造保持不变。
- OpenAI adapter 在 v0.3 可继续只接受 `TextPart`、`ImagePart`、`FunctionCallItem`、`FunctionResultItem`，对新增 part 抛 `InvalidItemError`。

### core.generation

`core.generation` 负责 full-context generation、streaming、reasoning config、structured output 与 remote context hint。

新增：

```text
RemoteContextHint
- previous_response_id
- covered_item_count
- cache_policy
- store
- expires_at
- provider_options
```

`RemoteContextHint` 只作为优化提示，不改变 Full-context First。`covered_item_count` 表示 `previous_response_id` 已覆盖完整 `GenerationRequest.items` 的前缀长度，用于 adapter 在 provider 请求层发送追加 suffix。

`GenerationRequest` 增加：

```text
remote_context: RemoteContextHint | None
```

兼容策略：

- `GenerationRequest.__init__` 新增 keyword-only `remote_context=None`。
- OpenAI adapter v0.3 可以先将 `remote_context` 映射为 OpenAI `previous_response_id` / `store` / `prompt_cache_*` 的子集，也可以明确不支持并要求用户继续用 `provider_options`；若映射，应加测试。
- 如果 `previous_response_id` 与 `covered_item_count` 均存在，OpenAI adapter 应只映射 `items[covered_item_count:]` 为 provider input。该差分传输是 OpenAI provider adapter 行为，不进入通用 `ReplayPolicy`。
- 如果 remote context 失效并触发 `on_remote_context_invalid="replay_without_remote_context"`，client 必须重新基于完整 `items` 构造请求，不能复用第一次差分请求 params 后简单删除 `previous_response_id`。
- v0.3 不要求新增 generation client 方法。

扩展 `ReasoningConfig`：

```text
mode: str | None
effort: str | None
budget_tokens: int | None
summary: str | None
include_trace: bool | None
provider_options: dict[str, Any]
```

`mode` 用于表达 on/off/auto/enabled/disabled/thinking 等跨 provider reasoning 开关语义。具体字符串暂不强枚举，避免过早锁死 provider 差异。

`ResponseFormat` 已在 v0.2 收敛为 JSON Schema structured output 专用模型，支持 schema name/description/schema/strict；v0.3 保持该结构，不暴露格式类型选择器，也不兼容 JSON mode / `json_object`。

## OpenAI Adapter 兼容边界

v0.3 的核心是扩展 core，不要求 OpenAI adapter 支持所有新增模型。为了保持行为清晰，OpenAI adapter 应采用以下策略：

- `MessageItem` 中的 `TextPart` 与 `ImagePart` 继续按 v0.2 映射。
- `AudioPart`、`VideoPart`、`FilePart` 暂不映射，抛 `InvalidItemError`，错误信息说明 OpenAI adapter 当前支持范围。
- `ReasoningItem` 作为 generation input 暂不映射，除非后续明确 provider 可安全 replay reasoning。
- OpenAI response 中 reasoning output 若 core 已有 `ReasoningItem`，可以从 metadata-only 策略升级为 normalized `ReasoningItem`；但若实现成本较高，可保留 v0.2 unsupported metadata 策略，留到 v0.4 adapter 映射时统一。
- `RemoteContextHint` 如映射 OpenAI `previous_response_id` / `store`，必须在用户文档说明这是 optimization hint；不得映射 OpenAI conversation 或其他 provider conversation 持久化能力。
- `RemoteContextHint.covered_item_count` 如与 `previous_response_id` 同时存在，OpenAI adapter 应将 provider input 优化为未覆盖 suffix；不得在覆盖边界缺失时猜测 history 与新增输入。
- `EmbeddingRequest.instructions` 对 OpenAI text embedding 暂不映射，除非 OpenAI embedding endpoint 明确支持；不确定时应抛 `UnsupportedCapabilityError` 或要求用户通过 `provider_options`。

### Remote context coverage tests

OpenAI adapter 需要覆盖：

- 存在 `previous_response_id` 且 `covered_item_count=N` 时，仅发送 `items[N:]`。
- 存在 `previous_response_id` 但缺少 `covered_item_count` 时，不猜测 history/append 边界；OpenAI adapter 应清晰失败，提示用户提供覆盖范围或不使用 `previous_response_id`。
- 差分传输时，provider-native snapshot replay 只要求 suffix items 满足策略；已覆盖 prefix 不参与本次 provider input 映射。
- `covered_item_count` 越界时抛清晰错误。
- previous response 失效且启用 fallback 时，第一次请求为 `previous_response_id + suffix input`，第二次请求为无 `previous_response_id + full input`。

## 风险与决策点

### ToolSpec 兼容形态

推荐实现为：

```text
FunctionToolSpec = dataclass(...)
ToolSpec = FunctionToolSpec
```

这样旧代码完全兼容，也能让新文档逐步推荐 `FunctionToolSpec`。如果未来需要保留独立类名用于 `isinstance`，可改为 `class ToolSpec(FunctionToolSpec)`，但 v0.3 不必复杂化。

### EmbeddingVector 兼容形态

`EmbeddingVector.embedding` 是 v0.2 公开字段，不能直接删除。推荐保留字段，并新增 `dense` / `sparse`：

- 构造时传 `embedding=[...]`，自动补 `dense`。
- 构造时传 `dense=[...]`，自动补 `embedding`。
- sparse-only provider 可以让 `embedding=None`，但这可能破坏旧类型假设。v0.3 应通过测试确认旧 OpenAI embedding 仍返回 list。

### RemoteContextHint 是否映射 OpenAI

OpenAI Responses API 支持 previous response/store/prompt cache 等远端状态能力。v0.3 只允许映射 `previous_response_id`、`store` 和 cache 相关 hint，不映射 provider conversation。这样可以保留 Full-context First，同时避免引入 provider conversation 持久化模型。

当 `previous_response_id` 有效且用户通过 `covered_item_count` 声明它覆盖完整 `items` 的前缀时，OpenAI adapter 应在 provider 请求层只发送追加 suffix。这是 OpenAI provider 的差分实现细节，不需要用户额外设置 transport 开关。这不改变用户传入完整上下文的要求；如果 response id 失效，fallback 必须回到完整 `items`。

### ReasoningItem replay policy

不同 provider 对 reasoning 是否可回传有安全和产品策略差异。`ReasoningItem.can_be_replayed` 默认应为 `False` 或 `None`，adapter 必须明确设置，不能假设 reasoning text 可以作为下一轮上下文。

### Capability 迁移节奏

API family capability 会让 capability 模型更完整，但不能马上删除旧布尔字段。v0.3 应先“双写”：新 family 字段用于新代码，旧字段继续服务 OpenAI adapter 与用户已有代码。

### Hosted/Remote Tools 暂缓

provider-hosted tools、remote tools 和 MCP 能力差异较大，当前不进入 v0.3 core。短期如确需调用 provider 专有 hosted tool，应通过 provider-specific `provider_options` 或后续 adapter 专有扩展处理，待语义稳定后再进入通用 core。
